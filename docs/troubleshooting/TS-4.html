mu-plugin 故障修复

troubleshooting L5

📖 What — 这是什么

mu-plugins（Must-Use Plugins，必须使用的插件）位于 wp-content/mu-plugins/ 目录。与普通插件不同，mu-plugins 有三个关键特征：

  • 自动激活 — 放进目录就生效，无需在后台激活
  • 无法从后台停用 — 后台没有"停用"按钮
  • 最先加载 — 在所有普通插件之前执行

这意味着 mu-plugin 的错误会在最早阶段触发，可能导致整个 WordPress 无法启动。

🎯 Why — 为什么要做

mu-plugin 故障特别棘手：后台可能完全无法访问，普通的"停用所有插件"也不会影响 mu-plugins。如果不知道 mu-plugins 的存在，可能会在常规排障中浪费大量时间。

常见场景：主机提供商通过 mu-plugin 注入功能（如缓存、安全规则），这些插件出问题时用户往往不知道它们的存在。

🔧 How — 怎么做

1. 检查 mu-plugins 目录

# 列出 mu-plugins
ls -la wp-content/mu-plugins/

# 用 WP-CLI 查看
locwp wp wp-train -- plugin list --status=must-use

2. 定位问题 mu-plugin

# 查看 debug.log 中的错误路径
grep "mu-plugins" wp-content/debug.log

如果日志指向 mu-plugins/problem-file.php，那就是它。

3. 修复方法

# 方法一：移除问题文件（最快）
mv wp-content/mu-plugins/problem-file.php wp-content/mu-plugins/problem-file.php.bak

# 方法二：移走整个 mu-plugins 目录
mv wp-content/mu-plugins wp-content/mu-plugins.bak
mkdir wp-content/mu-plugins

# 确认问题消失后，逐个移回排查
mv wp-content/mu-plugins.bak/good-file.php wp-content/mu-plugins/

4. 排障流程

  1. 查看 debug.log 是否指向 mu-plugins
  2. 如果不确定，先移走整个 mu-plugins 目录
  3. 问题消失 → 逐个移回文件确定罪魁祸首
  4. 问题依然 → 不是 mu-plugin 的锅，继续排查其他方向

⚠️ 常见问题与注意事项

  • mu-plugins 目录可能不存在——这是正常的，WordPress 不会自动创建它
  • mu-plugins 不支持子目录中的文件（除非有 loader），只有根目录下的 PHP 文件会被加载
  • 部分主机提供商的 mu-plugins 删除后会自动重建，需要联系主机解决
  • 恢复 mu-plugin 时注意文件权限，确保 Web 服务器可读
  • WP Reset 快照只恢复数据库，不会影响 mu-plugins 文件——这是好事也是限制
