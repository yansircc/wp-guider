查看 debug.log

故障排查 L3

📖 What — 这是什么

当 WordPress 的调试模式开启（WP_DEBUG 和 WP_DEBUG_LOG 设为 true）后，PHP 错误、警告和弃用提示会写入 wp-content/debug.log 文件。

这个日志文件是排障的第一手信息来源——白屏、功能异常、插件冲突的线索都记录在这里。读懂 debug.log 是从"看现象猜原因"进化到"看证据定根因"的关键技能。

🎯 Why — 为什么要做

WordPress 默认隐藏所有错误，遇到问题只会看到白屏或含糊的提示。debug.log 记录了真正的错误原因：哪个文件、哪一行、什么类型的错误。

没有日志的排障是盲人摸象——你可能花一小时逐个停用插件，而日志里一行就写明了是哪个插件的哪个函数出了问题。

🔧 How — 怎么做

查看日志

# 查看最新的 50 行日志
tail -50 wp-content/debug.log

# 实时监控（新错误会自动刷新）
tail -f wp-content/debug.log

# 搜索致命错误
grep "Fatal error" wp-content/debug.log

# 搜索特定插件的错误
grep "plugin-name" wp-content/debug.log

读懂日志条目

典型格式：

[30-Mar-2026 10:15:32 UTC] PHP Fatal error:
  Uncaught Error: Call to undefined function my_func()
  in /app/public/wp-content/plugins/bad-plugin/main.php on line 42

三个关键信息：

  • 错误级别 — Fatal error（致命，网站崩溃）、Warning（警告，可能影响功能）、Notice（提示，通常无害）
  • 错误描述 — 调用了不存在的函数 my_func()
  • 文件定位 — plugins/bad-plugin/main.php 第 42 行 → 问题出在 bad-plugin 插件

常用排查模式

  • 白屏：搜索 Fatal error，找到最近一条
  • 功能异常：搜索 Warning，按时间倒序查看
  • 定位问题插件：看文件路径中的 plugins/xxx 部分
  • 定位问题主题：看文件路径中的 themes/xxx 部分

⚠️ 常见问题与注意事项

  • debug.log 不存在：确认 WP_DEBUG 和 WP_DEBUG_LOG 都已设为 true，且 wp-content 目录有写入权限
  • 日志太大找不到重点：用 grep "Fatal" 过滤致命错误，或用 tail -20 只看最新的条目
  • Warning 不一定是 Bug：很多警告来自写法不规范的第三方插件，功能可能完全正常。优先关注 Fatal error
  • 日志持续增长：排障完成后关闭调试模式，或定期清空日志文件（: > wp-content/debug.log）
  • 生产环境安全：debug.log 可能包含敏感信息（文件路径、数据库查询），确保外部无法通过 URL 直接访问该文件
