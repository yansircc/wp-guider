综合故障注入 — 权限问题排查

故障排查 L5

📖 What — 这是什么

权限问题是 WordPress 最常见也最隐蔽的故障之一。它涉及两个层面：

  • 文件系统权限 — Linux/macOS 的文件/目录读写执行权限（chmod）。权限不正确会导致 WordPress 无法写入上传目录、无法更新插件/主题、无法生成缩略图等
  • WordPress 用户角色权限 — WordPress 内建的角色能力系统（Capabilities）。角色权限不当会导致用户无法执行应有的操作（如编辑者无法发布、作者无法上传媒体）

本练习通过故意制造权限问题，训练学员从现象出发定位根因的能力。

🎯 Why — 为什么要做

权限问题的症状往往与权限无关："图片上传失败"、"插件更新卡住"、"页面保存不了"——这些现象背后都可能是权限问题。能够识别并修复权限问题，是 WordPress 运维的核心技能。

通过故障注入（主动制造问题）来训练排查能力，比等真实故障发生时手忙脚乱要好得多。

🔧 How — 怎么做

场景一：文件系统权限故障

注入方式：修改 uploads 目录权限使其不可写

# 制造故障：移除 uploads 写入权限
chmod 555 wp-content/uploads

现象：上传图片时报错 "Unable to create directory" 或 "The uploaded file could not be moved"

排查思路：

  1. 看到上传错误 → 检查 debug.log 中的具体错误信息
  2. 日志指向 wp_mkdir_p() 或 move_uploaded_file() 失败
  3. 检查目录权限：ls -la wp-content/uploads
  4. 确认 Web 服务器用户（通常是 www-data）没有写入权限

修复：

# 恢复正确权限
chmod 755 wp-content/uploads

场景二：WordPress 角色权限故障

注入方式：将用户角色从"编辑"降为"投稿者"

现象：用户登录后看不到「页面」菜单，无法发布文章（只能提交待审）

排查思路：

  1. 用户报告"看不到某些功能" → 在 用户 → 所有用户 中检查该用户的角色
  2. 确认角色与预期不符
  3. 核对 WordPress 角色权限表：管理员 > 编辑 > 作者 > 投稿者 > 订阅者

修复：在用户编辑页面将角色改回正确值。

⚠️ 常见问题与注意事项

  • WordPress 标准目录权限：目录 755，文件 644。wp-config.php 建议 600 或 640
  • 不要用 777：chmod 777 虽然能解决所有权限报错，但会让任何用户都能读写执行，是严重的安全隐患
  • 所有者（Owner）也很重要：即使权限是 755，如果文件所有者不是 Web 服务器用户，也可能无法写入。用 ls -la 同时检查权限和所有者
  • WordPress 角色表：
    • 管理员 — 完全控制
    • 编辑 — 管理所有文章和页面
    • 作者 — 发布和管理自己的文章
    • 投稿者 — 写文章但不能发布
    • 订阅者 — 只能管理自己的资料
  • 排查黄金法则：先看日志（debug.log）→ 再检查文件权限（ls -la）→ 最后检查用户角色。不要凭猜测修改
